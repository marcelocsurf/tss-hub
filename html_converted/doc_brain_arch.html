This document captures all product decisions, validated architecture, and implementation specifications for TSS Brain. It serves as the single reference for continuing development across conversations. The Canon remains Tier 1 authority — this document is Tier 3 operational.

1. Roles & Permissions

Validated and deployed March 14, 2026. These are live in the app.

1.1 Admin

Full access. Creates coaches, manages system, sees all data. Currently: Marcelo Castellanos (marcelo@thesurfsequence.com).

1.2 Coordinator

Operations manager. Prepares everything before camp starts.

Can do:

  • Create new students (name, belt level, emergency contact)
  • Send intake link to students via WhatsApp or email
  • View and edit all student profiles
  • Create camp instances from templates
  • Assign students to camps (max 4 per camp)
  • Assign coaches to camps
  • Mark waiver as signed
  • View all data except admin system settings

Cannot do:

  • Change belt level (admin only)
  • Archive coaches
  • Modify camp templates
  • Access admin system settings

1.3 Coach

Executes sessions. Sees what they need to train the student well.

Can see:

  • Students assigned to their camps/sessions
  • Surf profile (stance, experience, board type, other sports, learning style)
  • Goals & barriers (short/mid/long term, fears)
  • Safety (allergies, injuries, swim level, emergency contact, medical notes)
  • Progression (belt, sequence, step, ocean level)
  • Session history of their students
  • Internal coach notes (their own)

Cannot see:

  • Contact details (email, phone)
  • Logistics (shirt size, how did they hear, waiver status)
  • Other coaches' internal notes
  • Intake link or intake status

1.4 Assistant

Helps coach in water. Sees absolute minimum for safety.

Can see:

  • Student name, belt level, ocean level
  • Allergies, injuries, swim level (safety-critical)
  • Emergency contact

Cannot see:

  • Goals, surf profile, session history, contact info, logistics, internal notes
  • Cannot edit anything

2. Operational Flow — Stage 1

2.1 New Student — Camp Flow

  • Booking confirmed outside app (Wix, WhatsApp, email)
  • Coordinator opens TSS Brain > Students > + Add Student
  • Fills minimum: first name, last name, belt level, emergency contact
  • Clicks Create Student
  • Copies Intake Link from student profile
  • Sends link to student via WhatsApp/email
  • Student opens link (no login needed), fills 5-step profile form
  • Student submits — data saved to their profile
  • Coordinator verifies "Profile completed" badge
  • Coordinator creates camp instance, assigns students (max 4), assigns coach
  • Coach sees student profile, goals, safety info when preparing sessions

2.2 Returning Student

  • Coordinator searches student in the system — they already exist
  • DO NOT create new record
  • Review and update if anything changed (injuries, goals)
  • Assign to new camp
  • New coach sees full history: what was worked, where left off, homework, previous coach notes

2.3 Camp Structure

  • 6 days / 6 sessions (Monday to Saturday typical)
  • Maximum 4 students per camp
  • Same coach ideally maintains all 6 days
  • If coach changes mid-camp, system shows continuity to new coach
  • Camp created from template (e.g., Novice 6-Day)

2.4 Session Flow — 3 Moments

Every session follows three distinct moments:

Moment 1 — Context: Who, where, when. Student selection, venue, conditions, risk assessment.

Moment 2 — Planning: The cascade. What to work on, with what drill, with what preparation.

Moment 3 — Close: Mandatory evaluation, feedback, homework, continuity, incident report.

2.5 Standalone Sessions

Less common but exists. Coach selects student (must already exist in system), same 3-moment flow without camp template.

4. Session Creation — The Cascade

Validated March 15, 2026. This is the core product decision for how sessions are built in the app.

Principle: The coach selects, not types. Each step filters the next. The system reflects TSS methodology logic — not generic forms.

4.1 Key Design Decision — Pilar Simplification

The 4 pillars (Technical, Tactical, Physical, Mental) + Safety exist doctrinally. But in the session creation flow, the coach does NOT select a pilar. Instead:

  • Technical + Tactical are fused — the coach goes directly to Pilar Part selection, filtered by student level. The system internally knows which pilar each part belongs to.
  • Mental is implicit — every session has a Mental Hack field. The mental pilar is embedded, not selected.
  • Physical is diagnostic — it appears in evaluation conclusions ("the limitation is physical"), not in session planning. The app identifies physical bottlenecks but doesn't prescribe physical solutions in Stage 1.
  • Safety is automatic — triggered by ocean rules when conditions exceed student level. Not a manual selection.

4.2 The Cascade — Step by Step

MOMENT 1 — CONTEXT

SAFETY OVERRIDE RULE

If ocean rules return "blocked" for the student's level + conditions, the session does NOT cancel. The system switches venue options to dry-land only (Surf skate, Gym, Tatami, Video Analysis Room) and filters drills to land-based only. The Pilar Part the coach selected is maintained — only the execution environment changes. No session is ever cancelled due to conditions.

7. Coach Invitation System

Deployed March 14, 2026. Admin-only feature.

7.1 Flow

  • Admin goes to Coaches > + Add Coach
  • Fills: name, email, phone, role, max belt permission, certification level
  • System sends invitation email
  • Coach clicks link in email
  • Redirected to Set Password page (/set-password)
  • Creates password, enters app with their role permissions

7.2 Current Workaround

If the email flow doesn't work, Admin sets password manually via Supabase SQL:

UPDATE auth.users SET encrypted_password = crypt('TempPassword!', gen_salt('bf')) WHERE email = 'coach@email.com';
